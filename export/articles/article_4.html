Using Claude Code with Ruby on Rails: Field Notes

原文來源：https://robbyonrails.com/articles/2026/03/12/field-notes-claude-code-in-a-rails-codebase/
以下為原文內容，英文術語後方附有中文對照（紫色）以輔助閱讀。

Over the past few months, a few of us at Planet Argon have been experimenting with Claude Code on real client Ruby on Rails （Ruby on Rails，一個流行的 Web 框架） projects. Not demos or greenfield apps （全新應用）, but the kind of long-lived codebases （長期維護的程式碼庫） where a backtrace （堆疊追蹤） can eat forty-five minutes before you even understand the call chain （呼叫鏈）.

Key finding: Some debugging sessions that used to take ~45 minutes now wrap up closer to 10 minutes. （核心發現：原本需要約 45 分鐘的除錯工作，現在約 10 分鐘即可完成。）

The authors are not ready to make big claims. But they've developed enough of a feel for when to reach for it, which model to use, and how to keep it from making a mess of your codebase （避免搞亂程式碼庫） that it felt worth writing down.

Key Topics Covered （涵蓋的主要主題）

  • Model selection （模型選擇）: Sonnet vs Opus and when to switch mid-session （何時在會話中途切換）
  • RSpec and Minitest workflows （測試框架工作流程）
  • Debugging with error monitoring payloads （利用錯誤監控數據進行除錯）
  • How to structure CLAUDE.md without turning it into a dumping ground （如何組織設定檔而不讓它變成垃圾場）

It's written for developers who are skeptical of the hype （對炒作持懷疑態度的開發者） but curious enough to run an experiment.

The engineers on your team who've already started experimenting with tools like this probably have a small head start （已有小小的先發優勢）. The gap closes quickly once you see where they actually help.

Why This Matters （為何重要）

This is one of the first practical, real-world assessments of Claude Code in a production Rails codebase （生產環境的 Rails 程式碼庫）—not a toy project, but actual client work with legacy code （遺留程式碼） and real debugging challenges. The measured 4-5x speedup in debugging sessions represents significant productivity gains for backend developers.

摘要

Planet Argon 團隊（由 Robby Russell 帶領）在真實 Rails 客戶專案中試用 Claude Code，非展示用途，而是長期維護的複雜既有程式庫，並分享實戰心得。

核心發現
45分 → 10分
除錯時間大幅縮短，生產力提升約 4 倍

這是在真實的 Rails 遺留程式庫中，而非全新專案，效果更令人信服。

實戰建議

模型選型策略

團隊分享何時應使用 Sonnet 與 Opus，以及在會話途中根據任務複雜度切換模型的時機與方法。

RSpec 與 Minitest 整合

說明如何與 Rails 測試框架整合，以及如何結合錯誤監控系統的 payload 進行高效除錯。

CLAUDE.md 結構設計

如何為專案建立有效的說明檔案，避免成為雜亂的知識轉儲，讓 Claude 更精準理解程式庫架構。

防止非預期修改

分享防止 Claude Code 在程式庫中進行意外修改的策略，在授予 AI 足夠彈性的同時保持控制。

目標讀者

本文明確寫給「對炒作持懷疑態度，但好奇到足以做實驗的開發者」，並指出團隊中的早期採用者確實享有先機，但這個優勢「一旦你看到 AI 真正有幫助的地方，很快就會消失」。

Summary

Planet Argon engineers used Claude Code on actual client Ruby on Rails projects — not demos or new apps, but established codebases with complex architectures — and documented their practical experiences.

Key Finding
45min → 10min
Debugging sessions dramatically shortened — ~4x productivity improvement

This was achieved in real legacy Rails codebases, making the result more credible than greenfield demos.

Recommendations

Model Selection Strategy

When to use Sonnet versus Opus, and how to switch models mid-session based on task complexity.

RSpec and Minitest Integration

Integration approaches with Rails test frameworks, and using error monitoring payloads for efficient debugging workflows.

CLAUDE.md Structure

How to structure project documentation files without creating disorganized knowledge dumps, so Claude better understands the codebase architecture.

Preventing Unintended Changes

Strategies to prevent Claude Code from making unexpected changes to the codebase while still giving it enough flexibility to be useful.

Target Audience

Written for "developers who are skeptical of the hype but curious enough to run an experiment." Early adopters on development teams gain an initial advantage that "closes quickly once you see where they actually help."

從 150 封未讀到零壓力：使用 MCP 和 GitHub Copilot 自動化收件匣分類

原文來源：devblogs.microsoft.com
以下為核心要點摘要，附英文技術詞彙對照（紫色）幫助理解。

核心摘要

作者利用 MCP（Model Context Protocol，模型上下文協議）搭配 WorkIQ 將 GitHub Copilot CLI（命令列介面）連接至 Microsoft 365 資料，打造一套自動化郵件與 Teams 訊息分類系統，解決每天面對 150+ 封未讀郵件的焦慮問題。

分類系統設計

  • 🔴 Action Required 需要直接行動的請求
  • 🟠 Review Needed 需要決策或時效性資訊
  • 🟢 Informational 僅供參考的 FYI 內容

解決的技術挑戰

  • Timeout 問題：大時間範圍查詢導致失敗。解法：chunked queries（分塊查詢），將多週掃描切分為 7 天區段，加入 graceful fallback（優雅降級）與重試機制
  • 重複項目問題：跨執行的重複訊息。解法：rolling JSONL index（滾動 JSONL 索引）搭配 semantic dedup（語意去重），比 fuzzy string matching（模糊字串比對）更準確地識別同意不同詞的訊息

關鍵學習

  • Prompt engineering（提示工程）的精確性對分類準確率至關重要
  • resilience patterns（韌性模式）（fallback、chunking）在 MCP 整合中不可或缺
  • Plain markdown + JSONL 檔案是輕量有效的 state management（狀態管理）方案

對程式設計師的意義

這個案例展示了 MCP 如何讓 AI assistant（AI 助理）突破對話框，直接操作 Microsoft 365 等企業系統。WorkIQ MCP 開源在 GitHub，可作為建構類似 productivity automation（生產力自動化）工具的起點，特別適合需要處理大量非同步溝通的工程師或 PM。

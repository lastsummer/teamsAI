MCP 2026 年度路線圖

原文來源：blog.modelcontextprotocol.io · 作者：David Soria Parra（Lead Maintainer）· 2026-03-09
以下為核心要點摘要，附英文技術詞彙對照（紫色）幫助理解。這是 MCP 官方發布的年度路線圖，所有使用或計劃採用 MCP 的開發者都應了解。

核心摘要

Model Context Protocol（模型上下文協議）自 2025 年 11 月發布正式規範以來，已從最初的本地工具連接器演進為企業級 agent 工作流程的基礎設施。本路線圖由首席維護者 David Soria Parra 發布，揭示 2026 年四大優先開發方向，並從「版本里程碑」模式轉向「優先領域框架」（priority-area framework）。

治理結構變化

MCP 開發不再以版本號為主軸，改由 Working Groups（工作組）主導，各組自行制定交付時程。SEP（Spec Enhancement Proposals，規範增強提案）是提交變更的正式管道，但只有與優先領域對齊的 SEP 才能快速推進。

四大優先領域

1. Transport Evolution and Scalability（傳輸層演進與擴展性）

現有 Streamable HTTP transport（可串流 HTTP 傳輸）無法支援生產環境的水平擴展——有狀態工作階段（stateful sessions）與負載平衡器（load balancers）衝突。解決方向：設計無狀態伺服器（stateless server）模式，以及透過 .well-known 端點實現能力探索（capability discovery），無需建立連線即可知曉伺服器功能。

2. Agent Communication（代理通訊）

Tasks primitive（任務原語）已實驗性發布，生產使用後暴露出兩個問題需完善：暫時性故障的重試機制（retry mechanics），以及任務結果的過期策略（result retention expiry policies）。這體現了 MCP 的迭代哲學：先發布實驗功能，收集生產數據，再精煉設計。

3. Governance Maturation（治理成熟化）

目前每個 SEP 都需要 Core Maintainers（核心維護者）全面審查，成為瓶頸。計劃建立貢獻者晉升路徑（contributor advancement pathway），並授權 Working Groups 在其領域內自行批准 SEP，Core Maintainers 保留戰略監督。

4. Enterprise Readiness（企業就緒性）

企業部署 MCP 時面臨四個一致性挑戰：稽核軌跡（audit trail）、SSO 整合認證（SSO-integrated authentication）、閘道行為規範（gateway behavior specifications）、設定可攜性（configuration portability）。此優先領域有意保持開放，邀請企業實踐者共同塑造方向，主要以協議擴展（protocol extensions）形式實現，不修改核心規範。

暫不納入本期的方向

  • Triggers and event-driven updates（觸發器與事件驅動更新）
  • Streamed and reference-based result types（串流與引用型結果類型）
  • Enhanced security and authorization（增強安全與授權機制）
  • Extensions ecosystem maturation（擴展生態系成熟化）

目前正在審查中的活躍提案：SEP-1932（DPoP）和 SEP-1933（Workload Identity Federation，工作負載身份聯合）。

對程式設計師的意義

這是 MCP 生態系最重要的戰略文件。如果你正在或計劃建立 MCP Server，需要特別注意：

  • 傳輸層將往無狀態方向設計，提前規劃架構可避免未來重構
  • 企業功能如 SSO、稽核即將標準化，商業產品可提前預留介面
  • 提交 SEP 或貢獻代碼，務必先確認與上述四大優先領域的對齊程度

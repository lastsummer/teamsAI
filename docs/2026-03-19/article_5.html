MCP Tool Design: Why Your AI Agent Is Failing (And How to Fix It)

原文來源：https://dev.to/aws-heroes/mcp-tool-design-why-your-ai-agent-is-failing-and-how-to-fix-it-40fc
以下為核心要點摘要，附英文技術詞彙對照（紫色）幫助理解。

核心摘要

MCP 伺服器失敗不是協議的問題，而是工具設計（tool design）的問題。GitHub 和 Block 等團隊通過減少工具數量和採用以結果為導向的設計（outcome-oriented design），獲得了可量化的改善。文章提出「能力三角框架（Capability Triangle）」，平衡三方：LLM（推理）、伺服器（運算）和領域專家（用戶上下文）。

關鍵設計原則

1. 工具數量：少即是多

  • 最佳 MCP 伺服器包含 5-15 個工具，超過 20 個工具後性能會崩潰
  • GitHub 將工具從 40 個削減至 13 個，基準測試（benchmark）顯著提升
  • 超過 20 個工具會導致上下文窗口膨脹（context window bloat）和工具幻覺（tool hallucination）

2. 以結果導向，而非 API 操作導向

  • 以 API 端點映射工具會製造高成本的多步驟呼叫鏈
  • 以結果導向的工具將業務邏輯封裝在伺服器端，降低 token（令牌）消耗、延遲和失敗點
  • 帕累托法則（Pareto principle）適用：20% 的 API 能力服務 80% 的用戶需求

3. 工具描述品質決定成敗

  • 97% 的品質問題來自工具描述不良
  • 六要素描述（用途、指引、限制、參數、長度、範例）可提升任務完成率 5.85 個百分點
  • 工具描述是 LLM 的主要決策介面；模糊描述導致選錯工具
✅ 實作建議：使用 JSON Schema（JSON 模式）定義參數類型安全（type safety），並在描述中提供至少一個具體使用範例。

對程式設計師的意義

這篇文章為建立 AI agent 整合的開發者提供了可直接執行的框架。能力三角框架（Capability Triangle）重新定義了以用戶需求而非 API 結構來設計工具。開發者可以立即應用這些基於證據的實踐：將工具限制在 5-15 個，按六要素撰寫描述，以結果而非操作來設計。這些改變對 Claude Code 技能設計同樣直接適用。

MCP Has No Security Layer. We Built One — MCPS.

原文來源：https://dev.to/razashariff/mcp-has-no-security-layer-we-built-one-mcps-5al2
以下為核心要點摘要，附英文技術詞彙對照（紫色）幫助理解。

核心摘要

MCP（Model Context Protocol）（模型上下文協議）已被 OpenAI、Google、Microsoft、AWS 採用，但它缺乏內建的安全層。38% 的 MCP 伺服器缺少身份驗證，60 天內已有 30 個 CVE（已知漏洞）被提報。MCPS 是一個托管安全代理（managed security proxy），通過單一 URL 提供訊息簽名、威脅偵測和稽核日誌。

⚠️ TLS 和 OAuth 不夠：「TLS 保護傳輸層...OAuth 驗證身份但不驗證訊息內容」，工具投毒（tool poisoning）和訊息篡改（message tampering）依然可行。

主要安全威脅

  • 工具投毒（Tool poisoning）：攻擊者篡改工具定義，注入惡意指令
  • Prompt 注入（Prompt injection）：透過工具回應插入惡意提示詞
  • 重放攻擊（Replay attacks）：重複使用已攔截的合法請求
  • 供應鏈攻擊（Supply chain attacks）：通過第三方 MCP 伺服器入侵

MCPS 解決方案

  • 涵蓋 OWASP MCP 十大風險中的 6 項，另有 4 項部分覆蓋
  • 使用 ECDSA P-256（橢圓曲線數位簽章演算法）對每條訊息進行加密簽名
  • 金鑰透過 GCP Cloud KMS（Google 雲端金鑰管理服務）管理並自動輪換
  • 執行期間掃描 SQL injection、命令注入（command injection）、路徑穿越（path traversal）
  • 免費方案：5 個 agent、每月 1,000 個請求、7 天稽核紀錄保留

對程式設計師的意義

任何在生產環境部署 MCP agent 的開發者都需要了解這個安全盲點。MCPS 的代理架構不需要修改現有 MCP 伺服器或客戶端程式碼，只需更換連線 URL 即可啟用完整的訊息完整性驗證（message integrity verification）和執行期威脅偵測。在處理敏感操作的 AI 系統中，這是不可忽視的安全基礎設施。

The Building Blocks of Claude Code: A Practical Guide for Founders（Claude Code 的建構基石：創辦人實用指南）

作者：Jean Galea｜2026-03-23｜原文來源：jeangalea.com
以下為核心要點摘要，附英文技術詞彙對照（紫色）幫助理解。

核心摘要

本文由創辦人 Jean Galea 撰寫，深度拆解 Claude Code 的客製化架構層。他將 Claude Code 定位為一個「可程式化的協作者」（programmable collaborator），而非單純的聊天機器人，並詳解七大核心建構基石，幫助技術創辦人打造真正整合到開發流程中的 AI 工作夥伴。

七大建構基石

1. CLAUDE.md — 持久性專案指令

這是一個在每次 session（工作階段） 開始時自動載入的「簡報文件」（briefing document）。內容包含：建置指令（build commands）、部署步驟（deployment steps）、程式碼偏好（code preferences），以及專案特定的限制條件。本質上是「任何你不想每次重複解釋的內容」。

2. Memory — 自動學習系統

不同於 CLAUDE.md 由使用者維護，memory files（記憶檔案） 是由 Claude 自己寫入的。這些檔案儲存在本地，讓 Claude 可以跨 session 記住偏好設定和專案的特殊慣例，無需每次重新說明。

3. Skills — 可重用工作流程

以 slash command（斜線指令） 觸發的程序，例如 /commit、/deploy，定義在帶有 YAML frontmatter（前置元資料） 的 Markdown 檔案中。作者指出 Skills 是「按需觸發」，相較於 CLAUDE.md 的「常駐載入」各有其使用時機。

4. Custom Agents — 專屬 AI 助理

具有獨立個性的專屬 AI，擁有自己的 system prompt（系統提示詞）、工具存取權限（tool access），以及模型選擇（model selection）。Agents 可以在背景獨立運行，並有細粒度的權限控制（scoped permissions）。

5. MCP Servers — 外部工具整合

Model Context Protocol（模型上下文協定） 將 Claude 連接到外部系統，例如 Playwright（瀏覽器自動化）、Notion、GitHub、Gmail。作者稱之為「能夠同時跨所有系統進行推理的協調層」（coordination layer）。

6. Hooks — 確定性規則執行

PreToolUse 和 PostToolUse hooks（鉤子） 會自動執行 shell 指令，繞過 Claude 的判斷，用於強制執行硬性安全規則（例如機械式阻擋危險指令）。這確保了某些規則是確定性的（deterministic），不依賴 AI 的解讀。

7. Settings & Permissions — 三層設定體系

由使用者層、專案層、本地層構成的三級階層（three-level hierarchy），支援對檔案和工具的細粒度允許/拒絕規則（granular allow/deny rules）。

核心洞察：複利效應

文章強調這些客製化層具有「複利效應」（compounding effect）——隨時間累積改善，使 Claude 從一個臨時工具逐漸演變為真正的整合團隊成員，「應該隨著與你合作的時間越長，表現越來越好」。

對程式設計師的意義

  • 架構思維：不要把 Claude Code 當聊天工具用，而要將其視為可程式化的系統，投資在 CLAUDE.md 和 Skills 的設定上。
  • 安全第一：善用 Hooks 執行不可繞過的安全規則，而非依賴 AI 判斷。
  • 長期投資：這些設定模式正成為業界標準，學習成本有長期回報。
  • Agents 分工：對不同任務建立專屬 Agent，比一個萬用 AI 更有效率且更安全。

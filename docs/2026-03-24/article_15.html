The Claude Code CVE That Should Change How You Review AI-Generated Code（Claude Code CVE 安全漏洞：應改變你審查 AI 生成程式碼的方式）

作者：Toni Antunovic｜原文來源：dev.to
以下為核心要點摘要，附英文技術詞彙對照（紫色）幫助理解。

核心摘要

本文揭露了兩個影響 Claude Code 的重大安全漏洞（CVE, Common Vulnerabilities and Exposures），並附上統計數據顯示 AI 生成程式碼的安全隱患有多嚴重——87% 的 PR 至少引入一個安全漏洞。文章呼籲開發者重新思考審查 AI 生成程式碼的方式。

兩大 CVE 詳情

CVE-2025-59536 CVSS 8.7 高危

⚠️ 遠端程式碼執行漏洞（Remote Code Execution），在 clone 含惡意程式碼的 repo 並啟動 Claude Code 後立即觸發
  • 攻擊向量：惡意的 .claude/settings.json 檔案
  • 機制：Pre-execution hooks（前置執行鉤子） 在信任提示（trust prompts） 出現前就已執行
  • 修補版本：1.0.111 及 2.0.65

CVE-2026-21852 CVSS 5.3 中危

⚠️ API 流量被靜默重定向至攻擊者控制的伺服器
  • 攻擊向量：利用 ANTHROPIC_BASE_URL 設定
  • 影響：完整的 authorization headers（授權標頭） 在信任對話框出現前即已傳送
  • 特別危險場景：共享的 Workspaces（工作區） 環境

驚人統計數據

DryRun Security 報告揭示：在 38 次掃描中，AI agents 共生成了 143 個安全問題，其中：

  • 87% 的 PR 至少引入一個安全漏洞（security vulnerability）
  • 常見問題包括：broken access control（存取控制破壞）、未驗證的 JWT handling（JWT 處理）、未認證的 endpoints（端點）

立即行動建議

  1. 立即更新 Claude Code 到已修補版本
  2. 若你的團隊曾存取不受信任的 repo，立即輪換（rotate） Anthropic API keys
  3. 將 .claude/settings.json 視為可執行程式碼，納入安全審查流程
  4. 在每個 PR 階段實施持續安全掃描（continuous security scanning），而非僅在最終審查時
  5. 部署本地優先的安全閘門（local-first security gates），在程式碼推送到遠端前先掃描

對程式設計師的意義

這兩個 CVE 揭示了一個根本性問題：AI 工具的設定檔本身就是攻擊面（attack surface）。當你 clone 一個 repo，其中的 .claude/settings.json 可以在你意識到之前就執行任意程式碼。

  • 不要盲目信任 clone 的 repo：在啟動任何 AI coding tool 前，先檢查設定檔
  • 安全審查前移：不能等到 code review 才做安全審查，要在每個 PR 都自動掃描
  • 87% 有漏洞這個數字是警鐘：AI 生成的程式碼不是「更安全的程式碼」，只是「更快產生的程式碼」
  • API key 管理：在 CI/CD 和共享環境中，特別留意環境變數的安全性
